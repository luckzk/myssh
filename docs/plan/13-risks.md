# 13 风险、合规与成本

复制一个堡垒机不是"抄页面"，真正的难点在协议、审计与合规。本章把坑提前摆出来。

## 13.1 技术风险

| 风险 | 影响 | 缓解 |
| --- | --- | --- |
| **自研图形协议（RDP/VNC）** | 工作量巨大、坑极多 | **复用 guacd**（已决策），不自研协议栈 |
| 高并发长连接稳定性 | 会话掉线、内存泄漏 | goroutine 生命周期管理、超时/心跳、压测 |
| 录像 I/O 成为瓶颈 | 拖慢交互 | 异步缓冲批量落盘、录像入对象存储 |
| 命令解析不准 | 审计漏记/误拦 | 行缓冲 + 终端转义处理；图形/交互式程序无法逐命令审计（已知局限） |
| 凭证泄露 | 严重安全事故 | 加密落库、KMS 管密钥、最小权限、连接时才解密、不回传明文 |
| guacd 暴露 | 被绕过堡垒机直连 | guacd 仅内网可达，不对外暴露 4822 |
| 强制下线/授权回收时效 | 越权访问 | 建链时再校验授权；控制信道即时下线 |

## 13.2 协议层难点细数

- **SSH**：PTY 尺寸同步、转义序列、SFTP 子系统、密钥/口令/键盘交互多种认证。
- **RDP（交给 guacd）**：NLA/CredSSP、剪贴板、驱动器/打印机重定向、分辨率自适应——这些都是 guacd 已解决的，自研代价极高。
- **数据库**：SQL 解析与改写、不同方言、高危语句识别。
- **K8s**：exec/attach TTY、多容器、权限边界。

## 13.3 合规要点（会话录制涉及隐私）

1. **告知与授权**：录屏/录命令需告知被监控方并取得授权，符合当地法律（如等保、个人信息保护）。
2. **审计不可篡改**：日志 append-only，可选哈希链 / 外发 SIEM。
3. **最小留存**：保留期可配置，到期归档/销毁。
4. **访问审计的审计**：查看录像本身也要鉴权并留痕。
5. **数据加密**：录像、凭证加密存储；传输全程 TLS。
6. **等保/分级保护**：如用于生产，需对照等保 2.0 三级相关要求设计。

## 13.4 法律与许可

- 原项目 **Next Terminal** 为开源项目（参考其仓库声明的许可证，通常 Apache-2.0）。复制其**架构与思路**用于自研是常见做法，但：
  - 不得直接照搬其受版权保护的**代码/商标/品牌**（"NEXT TERMINAL"、版权署名）。
  - 若直接基于其源码二次开发，须遵守其开源许可证义务。
  - **Apache Guacamole / guacd** 为 Apache-2.0，复用合规。
  - 探查到的 `license-api`（商业授权能力）属其商业版范畴，社区复制版应实现自有授权或置空。
- 建议：自研代码、自有品牌、明确 LICENSE，避免商标与版权纠纷。

## 13.5 成本估算（粗略）

| 项 | 估算 |
| --- | --- |
| 研发（2~3 全栈 × ~26 周） | 完整 44 模块的主要成本 |
| MVP（M1+M2，~10 周） | 可先验证产品形态 |
| 基础设施（小规模） | 1 台 4C8G 可跑 app+guacd+PG（数十并发会话） |
| 基础设施（规模化） | 按并发会话与录像存储线性增长（guacd CPU + 对象存储） |

## 13.6 不建议做的事

- ❌ 自研 RDP/VNC 协议栈（用 guacd）。
- ❌ 用纯 JWT 做会话（无法强制下线，堡垒机必须能即时踢人）。
- ❌ 把审计做成"事后补"（必须与会话同生命周期旁路）。
- ❌ 明文存凭证 / 把凭证下发到前端。
- ❌ 忽略合规告知就上线录屏。

## 13.7 结论

复用 guacd + Go 原生 SSH/DB/K8s 的混合方案，能在**可控成本**内复刻 Next Terminal 的全部 44 模块。关键投入应集中在：**会话网关的稳定性、审计管道的完整性、授权判定的严密性**——这三点决定产品是不是一台合格的堡垒机，而不是一个"带录屏的远程终端"。

按 [12 路线图](/plan/12-roadmap) 推进，M1+M2 即可得到可用 MVP，再逐步铺满。
